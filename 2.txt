Você é um lead dev sênior trabalhando no projeto PratoBy.

Contexto:
O PratoBy já tem:

* cardápio público por slug;
* checkout público;
* `createPublicOrder` server-side;
* OrdersPage;
* KDS/cozinha;
* painel de retirada;
* impressão de comanda;
* agendamento;
* pedido de balcão em desenvolvimento;
* página `/dashboard/qr-code` com QR normal e QR por mesa.

Agora implemente o MVP de **Pedido por Mesa via QR Code**.

Objetivo:
Quando o cliente escanear um QR de mesa, o cardápio público deve abrir já identificando a mesa. O pedido criado deve chegar para o restaurante como “Mesa X / Consumo no local”, sem exigir endereço ou retirada.

Não implementar nesta fase:

* fechamento de conta acumulada;
* mesa aberta com múltiplos pedidos agrupados;
* divisão de conta;
* taxa de serviço/couvert;
* pagamento online por mesa;
* garçom digital completo;
* nota fiscal;
* Firestore Rules frouxas;
* refactor amplo;
* dist/node_modules/docs.

====================================================

1. DETECTAR TOKEN DA MESA NA LOJA PÚBLICA
   ====================================================

Na loja pública, detectar token de mesa pela URL.

Aceitar:

* `?table=<token>`
* opcionalmente `?mesa=<token>` como alias
* se existir rota `/mesa/:token`, suportar também, se simples.

Ao abrir com token:

* buscar mesa ativa correspondente;
* validar que pertence à loja;
* se inválida/inativa:

  * mostrar aviso amigável;
  * manter cardápio navegável, mas sem modo mesa.

Não confiar apenas no frontend; backend deve validar token ao criar pedido.

Mostrar no topo/cardápio:
“Você está pedindo na Mesa 4”
ou:
“Consumo no local · Mesa 4”

====================================================
2. CHECKOUT EM MODO MESA
========================

Quando `tableToken` válido estiver presente:

O carrinho/checkout deve mudar para modo:

* `dine_in`
* consumo no local
* mesa identificada

Não pedir:

* endereço;
* bairro;
* CEP;
* taxa de entrega;
* escolha entrega/retirada.

Mostrar:

* Mesa X;
* consumo no local;
* aviso:
  “Seu pedido será enviado para a cozinha com esta mesa.”

Pagamento:

* para MVP, permitir pagamento presencial/manual conforme loja:

  * pagar no caixa;
  * dinheiro;
  * Pix manual;
  * cartão/maquininha;
* se Asaas online estiver ativo, não precisa integrar pagamento online por mesa neste patch, a menos que já funcione naturalmente.
* Não exigir pagamento online.

Payload para `createPublicOrder` deve incluir:

```js
orderType: "table"
fulfillmentType: "dine_in"
source: "table_qr"
tableToken
```

Backend deve resolver:

```js
tableId
tableLabel
tableNumber
```

Não permitir o client mandar livremente “Mesa 4” sem token validado.

====================================================
3. BACKEND CREATEPUBLICORDER
============================

Atualizar `createPublicOrder` para aceitar pedido de mesa.

Regras:

* se `orderType === "table"` ou `fulfillmentType === "dine_in"`:

  * exigir `tableToken`;
  * validar loja pública;
  * buscar mesa ativa da loja pelo token;
  * se inválido/inativo, rejeitar;
  * não exigir endereço;
  * não calcular taxa de entrega;
  * fulfillment vira consumo no local;
  * salvar `tableId`, `tableLabel`, `tableNumber`, `tableTokenHash` se necessário.
* backend continua recalculando preços/cupom/agendamento.
* não aceitar mesa de outra loja.
* pedido por mesa deve nascer com status normal:

  * `pendente` ou `confirmado`, conforme fluxo atual de pedido público.
* não adicionar status novo.

Campos sugeridos no pedido:

```js
source: "table_qr",
orderType: "table",
fulfillmentType: "dine_in",
isTableOrder: true,
tableId,
tableLabel,
tableNumber,
tableToken,
deliveryType: "dine_in",
acceptDelivery: false,
isDelivery: false
```

Se salvar token completo for sensível, salvar token parcial/hash e `tableId`.

====================================================
4. ORDERS PAGE
==============

Mostrar pedidos de mesa claramente.

No card/lista:

* badge “Mesa”
* `Mesa 4`
* “Consumo no local”

Não mostrar endereço vazio.
Não confundir com “Retirada”.

No card expandido:

* bloco compacto:
  “Mesa 4”
  “Consumo no local”
* ações de status seguem fluxo normal.

WhatsApp:

* se cliente informou telefone, permitir;
* se não informou, esconder/desabilitar.

Comanda:

* mostrar:
  “Mesa 4”
  “Consumo no local”
* não mostrar entrega/retirada.

====================================================
5. KDS / COZINHA
================

KDS deve exibir pedido por mesa como operacional normal.

Mostrar no card:

* “Mesa 4”
* badge “Mesa”

Se status:

* confirmado/preparando/pronto → aparece normalmente.

Não mexer profundamente no KDS se o fluxo atual já mostra.

====================================================
6. CUSTOMER DISPLAY / PAINEL DE RETIRADA
========================================

Pedido por mesa não necessariamente precisa aparecer no painel de retirada.

Regra:

* Se painel de retirada é para retirada/balcão, pedido de mesa pode ficar fora.
* Se aparecer, deve ser sem PII e com “Mesa 4”.
* Decida o comportamento mais seguro e documente no resumo.

Minha recomendação:

* não exibir mesa no painel de retirada público por padrão.
* KDS/OrdersPage já resolvem a operação.

====================================================
7. TRACKING DO CLIENTE
======================

Se o cliente finalizar pedido de mesa:

* tracking deve mostrar:
  “Pedido enviado para a Mesa 4”
  “Acompanhe o preparo pelo restaurante.”
* não mostrar endereço/entrega.
* manter status normal.

Se cliente voltar ao cardápio pelo QR:

* pode continuar fazendo novo pedido para mesma mesa.
* V1 não agrupa em conta única.

====================================================
8. FECHAMENTO DE CONTA — NÃO IMPLEMENTAR
========================================

Não implementar conta acumulada ainda.

Deixar TODO/estrutura para V2:

* sessão de mesa;
* múltiplos pedidos agrupados;
* solicitar fechamento;
* total da mesa;
* fechar mesa.

Mas neste MVP:

* cada pedido da mesa é um pedido separado.
* fechamento de conta é manual pelo restaurante.
